# History & Undo System

## Overview

Every change the user makes to their trip plan is tracked. The history system provides:

1. **Undo / Redo** — Ctrl+Z to undo, Ctrl+Shift+Z to redo (works from any section)
2. **Change log** — A visible, chronological list of all changes with human-readable descriptions
3. **Revert** — Click any entry in the change log to revert to that point in time

This is built on **Immer patches** integrated with **Zustand**, giving us automatic change tracking with zero manual bookkeeping.

---

## How It Works: Immer Patches

### The Concept

Immer is a library that lets you write "mutable" code that produces immutable state updates. As a side effect, it can generate **patches** — small objects that describe exactly what changed.

```typescript
// When you do this:
state.places.push({ id: 'abc', name: 'Ichiran Ramen', ... })

// Immer generates a patch like this:
{
  op: 'add',
  path: ['places', 5],
  value: { id: 'abc', name: 'Ichiran Ramen', ... }
}

// And an inverse patch:
{
  op: 'remove',
  path: ['places', 5]
}
```

The **patch** describes how to go forward (apply the change).
The **inverse patch** describes how to go backward (undo the change).

This is the foundation of the entire history system.

## Architecture

### Zustand + Immer Middleware

The trip store uses Zustand's Immer middleware. Every state mutation automatically produces patches.

```typescript
import { create } from 'zustand'
import { immer } from 'zustand/middleware/immer'
import { produceWithPatches, enablePatches, type Patch } from 'immer'

// Enable Immer's patch generation
enablePatches()
```

### History Store

A separate Zustand store manages the undo/redo stack and change log.

```typescript
interface HistoryEntry {
  id: string                      // nanoid
  timestamp: string               // ISO timestamp
  description: string             // "Added Ichiran Ramen to Ramen Spots"
  patches: Patch[]                // Forward patches (to redo)
  inversePatches: Patch[]         // Inverse patches (to undo)
}

interface HistoryState {
  // The change log (all entries, for display in History section)
  entries: HistoryEntry[]

  // Undo/redo position
  // Points to the current position in the entries array
  // Entries after this index are "future" (available for redo)
  currentIndex: number

  // Actions
  pushEntry: (entry: Omit<HistoryEntry, 'id' | 'timestamp'>) => void
  undo: () => void
  redo: () => void
  revertTo: (entryId: string) => void
  canUndo: () => boolean
  canRedo: () => boolean
}
```

### How a Change Flows

```
User action (e.g., adds a place)
  │
  ▼
Trip store mutation (Zustand + Immer)
  │
  ├── State updates immediately
  │
  ├── Immer generates patches + inverse patches
  │
  ▼
History store receives:
  ├── patches (to redo this action)
  ├── inversePatches (to undo this action)
  └── description (human-readable, generated by the action)
  │
  ▼
Entry pushed to history stack
  │
  ▼
Persisted to IndexedDB (optional, for surviving page refresh)
```

## Implementation

### Wrapping Store Actions with Patch Tracking

Every trip store action that modifies state goes through a helper that captures patches:

```typescript
import { produceWithPatches, type Patch } from 'immer'

// Helper to execute a state mutation and capture patches
function withHistory(
  description: string,
  recipe: (draft: TripState) => void
) {
  const currentState = useTripStore.getState()

  const [nextState, patches, inversePatches] = produceWithPatches(
    currentState,
    recipe
  )

  // Apply the state change
  useTripStore.setState(nextState)

  // Record in history
  useHistoryStore.getState().pushEntry({
    description,
    patches,
    inversePatches
  })

  // Persist to DB
  syncToDatabase(nextState)
}

// Usage in store actions:
addPlace: (place: Place) => {
  withHistory(
    `Added "${place.name}" to ${getListName(place.listIds[0]) || 'Places'}`,
    (draft) => {
      draft.places.push(place)
    }
  )
}

movePlace: (placeId: string, fromDay: number | null, toDay: number) => {
  const place = getPlace(placeId)
  withHistory(
    `Moved "${place.name}" from ${formatDay(fromDay)} to Day ${toDay + 1}`,
    (draft) => {
      const p = draft.places.find(p => p.id === placeId)
      if (p) p.dayIndex = toDay
    }
  )
}
```

### Undo Implementation

```typescript
undo: () => {
  const { entries, currentIndex } = get()
  if (currentIndex < 0) return // Nothing to undo

  const entry = entries[currentIndex]
  const tripState = useTripStore.getState()

  // Apply inverse patches to go backward
  const previousState = applyPatches(tripState, entry.inversePatches)
  useTripStore.setState(previousState)

  // Move the pointer back
  set({ currentIndex: currentIndex - 1 })

  // Persist
  syncToDatabase(previousState)
}
```

### Redo Implementation

```typescript
redo: () => {
  const { entries, currentIndex } = get()
  if (currentIndex >= entries.length - 1) return // Nothing to redo

  const entry = entries[currentIndex + 1]
  const tripState = useTripStore.getState()

  // Apply forward patches to go forward
  const nextState = applyPatches(tripState, entry.patches)
  useTripStore.setState(nextState)

  // Move the pointer forward
  set({ currentIndex: currentIndex + 1 })

  // Persist
  syncToDatabase(nextState)
}
```

### Revert to Point

```typescript
revertTo: (entryId: string) => {
  const { entries, currentIndex } = get()
  const targetIndex = entries.findIndex(e => e.id === entryId)

  if (targetIndex < 0 || targetIndex >= currentIndex) return

  let state = useTripStore.getState()

  // Apply inverse patches from currentIndex down to targetIndex (inclusive)
  for (let i = currentIndex; i > targetIndex; i--) {
    state = applyPatches(state, entries[i].inversePatches)
  }

  useTripStore.setState(state)
  set({ currentIndex: targetIndex })
  syncToDatabase(state)
}
```

### Branch Behavior (New Change After Undo)

When the user undoes some actions and then makes a new change, the "future" entries (redoable entries) are discarded. This is how text editors and most undo systems work.

```typescript
pushEntry: (entry) => {
  const { entries, currentIndex } = get()

  // Discard any entries after the current position (the "future")
  const newEntries = entries.slice(0, currentIndex + 1)

  // Add the new entry
  newEntries.push({
    ...entry,
    id: nanoid(),
    timestamp: new Date().toISOString()
  })

  set({
    entries: newEntries,
    currentIndex: newEntries.length - 1
  })
}
```

## Keyboard Shortcuts

### Global (work from any section)

| Shortcut | Action |
|----------|--------|
| Ctrl/Cmd + Z | Undo last change |
| Ctrl/Cmd + Shift + Z | Redo last undone change |
| Ctrl/Cmd + Y | Redo (alternative shortcut) |

### Implementation

```typescript
// Global keyboard listener (in App.tsx or a dedicated hook)
useEffect(() => {
  const handleKeyDown = (e: KeyboardEvent) => {
    const mod = e.metaKey || e.ctrlKey

    if (mod && e.key === 'z' && !e.shiftKey) {
      e.preventDefault()
      useHistoryStore.getState().undo()
      showToast('Undid: ' + getLastDescription())
    }

    if (mod && (e.key === 'z' && e.shiftKey) || (e.key === 'y')) {
      e.preventDefault()
      useHistoryStore.getState().redo()
      showToast('Redid: ' + getNextDescription())
    }
  }

  window.addEventListener('keydown', handleKeyDown)
  return () => window.removeEventListener('keydown', handleKeyDown)
}, [])
```

## Persistence

### History entries in IndexedDB

History entries are persisted to survive page refreshes:

```typescript
// Dexie table
historyEntries: 'id, timestamp'

// On each pushEntry, also write to DB
// On app startup, load history from DB and hydrate the history store
// Cap at 500 entries (remove oldest when exceeded)
```

### History size limits

- **Max entries**: 500 (keeps the last 500 changes)
- **When exceeded**: Remove the oldest entries (FIFO)
- **Patch size**: Most patches are small (a few hundred bytes). 500 entries ≈ ~500KB worst case.
- **Clear history**: Option in settings to clear all history

**Visual indicators:**
- Entries above the current position are "active" (applied changes)
- Entries below the current position are "undone" (available for redo, shown muted)
- A visual separator marks the current position
- "Revert" button appears on hover for each active entry
- Clicking "Revert" on entry X undoes all entries from the current position down to X

## Edge Cases

### Undo after import
- Importing a trip creates a single "Imported trip from JSON" history entry
- Undoing it restores the entire previous state
- This works because the import patches capture the complete state transition

### Multiple rapid changes
- Each distinct action gets its own entry (no debouncing)
- Drag-and-drop reordering creates one entry per drop (not per pixel moved)

### Conflicting undo
- If a place is deleted, then the user undoes a previous action that referenced that place:
  - The undo uses inverse patches which restore the exact state, so deleted items come back
  - This is inherently safe because patches operate on the full state, not individual entities

### History after clear/reset
- If the user clears their trip and starts fresh, history is also cleared
- The "new trip" starts with a clean history

---

## What This System Does NOT Do

- **Named snapshots**: No "save as version 1.0" capability. This is a linear undo stack, not a version manager. (Could be added in the future.)
- **Selective undo**: Cannot undo a specific past action without also undoing everything after it. (Standard undo behavior.)
- **Diff visualization**: Does not show a visual diff between two states. The change log is text-only. (Could be added.)
- **Collaboration conflict resolution**: Single-user system. No merge conflicts to resolve.
